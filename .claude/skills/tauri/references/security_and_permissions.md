## Capabilities

Capabilities map permissions to windows/webviews, enabling fine-grained security control. All files in `src-tauri/capabilities/` auto-enable unless explicitly configured in `tauri.conf.json`.

### Defining Capabilities

**As separate files** (`src-tauri/capabilities/<identifier>.json` or `.toml`):

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "main-capability",
  "description": "Capability for the main window",
  "windows": ["main"],
  "permissions": [
    "core:path:default",
    "core:event:default",
    "core:window:default",
    "core:app:default",
    "core:resources:default",
    "core:menu:default",
    "core:tray:default",
    "core:window:allow-set-title"
  ]
}
```

**Referenced in tauri.conf.json:**

```json
{
  "app": {
    "security": {
      "capabilities": ["my-capability", "main-capability"]
    }
  }
}
```

**Inline in tauri.conf.json:**

```json
{
  "app": {
    "security": {
      "capabilities": [
        {
          "identifier": "my-capability",
          "description": "Used for all windows",
          "windows": ["*"],
          "permissions": ["fs:default", "allow-home-read-extended"]
        },
        "my-second-capability"
      ]
    }
  }
}
```

### Platform-Specific Capabilities

Restrict to specific platforms with the `platforms` array (`linux`, `macOS`, `windows`, `iOS`, `android`):

```json
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "desktop-capability",
  "windows": ["main"],
  "platforms": ["linux", "macOS", "windows"],
  "permissions": ["global-shortcut:allow-register"]
}
```

### Remote API Access

Grant Tauri command access to remote URLs (e.g., for embedded web content):

```json
{
  "$schema": "../gen/schemas/remote-schema.json",
  "identifier": "remote-tag-capability",
  "windows": ["main"],
  "remote": {
    "urls": ["https://*.tauri.app"]
  },
  "platforms": ["iOS", "android"],
  "permissions": ["nfc:allow-scan", "barcode-scanner:allow-scan"]
}
```

**Note:** Linux and Android cannot distinguish between `<iframe>` requests and window requests.

### Command Access Control in build.rs

Register commands explicitly to restrict availability:

```rust
fn main() {
    tauri_build::try_build(
        tauri_build::Attributes::new()
            .app_manifest(tauri_build::AppManifest::new().commands(&["your_command"])),
    )
    .unwrap();
}
```

### Key Rules

- Security boundaries depend on window **labels** (not titles).
- Windows/webviews in multiple capabilities **merge** their security boundaries.
- Schemas auto-generated by `tauri-build`: use `desktop-schema.json` or `mobile-schema.json`.

> Source: <https://v2.tauri.app/security/capabilities/>

## Permissions

Permissions describe explicit privileges for commands. They control which frontend operations can access backend functionality.

### Identifier Format

- `<name>:default` -- Default plugin/application permission
- `<name>:<command-name>` -- Individual command permission
- ASCII lowercase `[a-z]` only, max 116 characters
- Plugin prefix `tauri-plugin-` is prepended automatically at compile time

### File Locations

| Context | Path | Format |
|---------|------|--------|
| Plugin permissions | `permissions/<identifier>.toml` | TOML only |
| App permissions | `src-tauri/permissions/<identifier>.toml` | TOML only |
| App capabilities | `src-tauri/capabilities/<identifier>.json/.toml` | JSON, JSON5, or TOML |

### Defining Permissions (TOML)

**Scope-based permission:**

```toml
[[permission]]
identifier = "scope-home"
description = "Permits access to all files and list content in $HOME."

[[scope.allow]]
path = "$HOME/*"

[[scope.deny]]
path = "$HOME/secret"
```

**Command-based permission:**

```toml
[[permission]]
identifier = "read-files"
description = "Enables file read commands without pre-configured paths."
commands.allow = [
    "read_file",
    "read",
    "open",
    "read_text_file",
    "read_text_file_lines",
    "read_text_file_lines_next"
]
```

**Single command permission:**

```toml
[[permission]]
identifier = "allow-mkdir"
description = "Enables the mkdir command."
commands.allow = ["mkdir"]
```

### Permission Sets

Group multiple permissions under one identifier:

```toml
[[set]]
identifier = "allow-home-read-extended"
description = "Allows non-recursive read access and directory creation."
permissions = [
    "fs:read-files",
    "fs:scope-home",
    "fs:allow-mkdir"
]
```

### Key Rules

- Permissions must be referenced in a capability to take effect.
- Default permissions are auto-added when using the Tauri CLI to add plugins.
- App developers can extend plugin permissions or define custom ones.

> Source: <https://v2.tauri.app/security/permissions/>

## Scopes

A scope is a granular way to define allowed/disallowed behavior of a Tauri command. Scope types must be `serde`-serializable. **Deny always takes precedence over allow.**

### Allow and Deny

| Category | Behavior |
|----------|----------|
| `allow` | Permits specified operations |
| `deny` | Prohibits specified operations (supersedes allow) |

### Filesystem Plugin Example

```toml
[[permission]]
identifier = "scope-applocaldata-recursive"
description = "Recursive access to $APPLOCALDATA folder"

[[permission.scope.allow]]
path = "$APPLOCALDATA/**"
```

### Scope Composition

**Deny defaults (block dangerous paths):**

```toml
[[set]]
identifier = "deny-default"
description = "Denies dangerous Tauri-relevant files"
permissions = ["deny-webview-data-linux", "deny-webview-data-windows"]
```

**Combined scope (allow + deny):**

```toml
[[set]]
identifier = "scope-applocaldata-reasonable"
permissions = ["scope-applocaldata-recursive", "deny-default"]
```

**Command-specific scope:**

```toml
[[set]]
identifier = "read-files-applocaldata"
permissions = ["scope-applocaldata-reasonable", "allow-read-file"]
```

### Key Rules

- Each plugin defines its own scope types.
- Custom commands require manual scope definition and enforcement.
- Command developers must audit for scope bypasses.
- Scopes can apply globally (plugin-wide) or per-command.

> Source: <https://v2.tauri.app/security/scope/>

## Content Security Policy (CSP)

Tauri uses CSP to mitigate XSS attacks. Local scripts are hashed at compile time; external scripts/styles are protected via cryptographic nonces.

### Configuration

Set in `tauri.conf.json` under `app.security.csp`:

```json
{
  "app": {
    "security": {
      "csp": {
        "default-src": "'self' customprotocol: asset:",
        "connect-src": "ipc: http://ipc.localhost",
        "font-src": ["https://fonts.gstatic.com"],
        "img-src": "'self' asset: http://asset.localhost blob: data:",
        "style-src": "'unsafe-inline' 'self' https://fonts.googleapis.com"
      }
    }
  }
}
```

### WebAssembly

When using WASM (e.g., Rust-based frontends), add `'wasm-unsafe-eval'` to `script-src`.

### Best Practices

- Restrict sources to trusted hosts you control.
- Avoid CDN-hosted scripts and remote content.
- Tauri injects nonces/hashes into bundled assets automatically during compilation.
- Refer to MDN docs for [`script-src`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/script-src) and [`style-src`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Security-Policy/style-src) directives.

> Source: <https://v2.tauri.app/security/csp/>

## Runtime Authority

The runtime authority is the core Tauri component that enforces the ACL system at runtime, sitting between the webview IPC layer and command handlers.

### Command Invocation Flow

1. Webview sends an invoke request.
2. Runtime authority validates the origin is permitted for the requested command.
3. Verifies the origin is included in applicable capabilities.
4. Injects scopes into the invoke request if defined.
5. Passes the authorized request to the command handler.

If the origin is **not allowed**, the request is denied and the command handler is **never invoked**.

### Security Boundaries

**Protections provided:**
- Minimizes frontend compromise impact.
- Prevents/reduces local system interface exposure.
- Prevents/reduces privilege escalation (frontend to backend).

**Not protected against:**
- Malicious or compromised Rust code.
- Overly permissive scopes.
- Missing scope checks in custom commands.
- Intentional Rust-side bypasses.
- WebView zero-days or unpatched vulnerabilities.
- Supply chain attacks.

> Source: <https://v2.tauri.app/security/runtime-authority/>
